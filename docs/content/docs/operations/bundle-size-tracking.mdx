# Bundle Size Tracking

This document describes the bundle size monitoring system for the next-ship repository.

## Overview

Bundle size directly impacts user experience through load times and data usage. This system monitors bundle size automatically and alerts when limits are exceeded.

## How It Works

### Automatic Monitoring

The `bundle-size-tracking.yml` workflow runs on:
- **Every push to main**
- **Every pull request**
- **Weekly on Sundays at 4:00 AM AWST** (8:00 PM UTC Saturday)

### What Gets Measured

1. **First Load JS** - Critical JavaScript for initial page load
2. **Total JS Chunks** - Number of code-split chunks
3. **Chunk Sizes** - Individual chunk file sizes
4. **Static Assets** - Total size of all static files

### Thresholds

Current limits (warning only, not blocking):

| Metric | Limit | Status |
|--------|-------|--------|
| First Load JS | 200 KB | Warning if exceeded |
| JS Chunks | 100 | Warning if exceeded |
| Individual Chunk | 250 KB | Warning if exceeded |

## Optimizing Bundle Size

### Strategies

1. **Code Splitting**
   ```typescript
   // Use dynamic imports for large components
   const HeavyComponent = dynamic(() => import('./HeavyComponent'));
   ```

2. **Tree Shaking**
   - Ensure proper ESM imports
   - Avoid importing entire libraries
   ```typescript
   // Good
   import { specificFunction } from 'library';
   
   // Bad
   import * as library from 'library';
   ```

3. **Dependency Analysis**
   ```bash
   # Check bundle composition
   cd apps/web
   ANALYZE=true pnpm build
   ```

4. **Lazy Loading**
   - Load non-critical components on demand
   - Use intersection observer for below-fold content

### Common Issues

**Large vendor bundle:**
- Solution: Check for duplicate dependencies across packages
- Use `pnpm why <package>` to find duplicates

**Unused imports:**
- Solution: Run `pnpm dead-code` to find unused exports

**Large images/assets:**
- Solution: Use Next.js Image component with proper sizing

## Best Practices

1. **Check on every PR**: Review bundle size changes before merging
2. **Lazy load**: Use dynamic imports for non-critical features
3. **Tree shake**: Ensure proper imports for optimal dead code elimination
4. **Monitor dependencies**: Regularly audit for unused/bloated packages
5. **Split wisely**: Code split at route boundaries, not arbitrarily

## FAQ

**Q: Does this block PRs if bundle is too large?**
A: No, currently warnings only. We may add blocking limits for extreme cases.

**Q: Can I ignore the warnings?**
A: Yes, but please document why in the PR description if the increase is intentional.

**Q: How do I reduce bundle size?**
A: Start with the bundle analyzer to identify the largest modules, then:
1. Replace with lighter alternatives
2. Use dynamic imports
3. Remove unused code

**Q: What's a good First Load JS target?**
A: Under 200KB is ideal. Under 300KB is acceptable. Over 500KB needs attention.
