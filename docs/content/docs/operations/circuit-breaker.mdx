# Circuit Breaker Pattern

## Overview

Circuit breakers prevent cascading failures when external services become unavailable or degraded. They act as a safety switch that "opens" when failures exceed a threshold, failing fast instead of overwhelming the struggling service.

## How It Works

### State Machine

```
CLOSED (normal operation)
    ↓ 5 failures
OPEN (failing fast)
    ↓ 30 seconds
HALF_OPEN (testing recovery)
    ↓ 3 successes
CLOSED (back to normal)
```

**CLOSED**: Requests pass through normally
**OPEN**: Requests fail immediately without calling the service
**HALF_OPEN**: Testing if service has recovered with limited traffic

### Benefits

- **Fail Fast**: No waiting for timeouts when service is down
- **Prevent Cascading**: Protects your app from overwhelming failing services
- **Auto-Recovery**: Automatically tests service recovery
- **Monitoring**: Track service health and failure rates

## Custom Circuit Breakers

### Basic Usage

```typescript
import { createCircuitBreaker } from "@repo/circuit-breaker";

const myBreaker = createCircuitBreaker("my-service", {
  failureThreshold: 5,
  resetTimeout: 30000,
  successThreshold: 3,
});

// Use it
try {
  const result = await myBreaker.execute(async () => {
    return await fetchExternalAPI();
  });
} catch (error) {
  if (isCircuitBreakerOpenError(error)) {
    // Circuit is open, service is down
    // Use fallback or cache
  }
}
```

### Advanced Configuration

```typescript
import { CircuitBreaker } from "@repo/circuit-breaker";

const breaker = new CircuitBreaker({
  serviceName: "external-api",
  failureThreshold: 5,
  resetTimeout: 30000,
  successThreshold: 3,
  isFailure: (error) => {
    // Custom logic to determine if error counts
    if (error instanceof Error) {
      // Don't count 404s as failures
      return !error.message.includes("404");
    }
    return true;
  },
  onStateChange: (state, service) => {
    // Alert when circuit opens
    console.log(`${service} circuit is now ${state}`);
  },
});
```

## Integration Patterns

### With Middleware

```typescript
import { withCircuitBreaker } from "@repo/circuit-breaker";

const safeGetUser = withCircuitBreaker(
  async (userId: string) => clerkClient.users.getUser(userId),
  clerkCircuitBreaker
);

// Usage
const user = await safeGetUser("user_123");
```

### With React Server Components

```typescript
import { clerkCircuitBreaker } from "@repo/circuit-breaker";

export default async function UserProfile({ userId }: { userId: string }) {
  try {
    const user = await clerkCircuitBreaker.execute(async () => {
      return await clerkClient.users.getUser(userId);
    });
    
    return <div>{user.firstName}</div>;
  } catch (error) {
    if (isCircuitBreakerOpenError(error)) {
      return <div>Authentication temporarily unavailable</div>;
    }
    throw error;
  }
}
```

### With Error Boundaries

```typescript
import { isCircuitBreakerOpenError } from "@repo/circuit-breaker";

export default function App() {
  return (
    <ErrorBoundary
      fallback={(error) => {
        if (isCircuitBreakerOpenError(error)) {
          return <ServiceDownPage />;
        }
        return <GenericErrorPage />;
      }}
    >
      <YourApp />
    </ErrorBoundary>
  );
}
```

## Best Practices

### 1. Choose Appropriate Thresholds

**Critical Services** (Auth, Payments):
- Low failure threshold (3-5)
- Fast reset (15-30 seconds)

**Non-Critical Services** (Analytics, CMS):
- Higher threshold (10-20)
- Slower reset (60+ seconds)

### 2. Always Handle CircuitOpenError

```typescript
try {
  await breaker.execute(() => fetchData());
} catch (error) {
  if (isCircuitBreakerOpenError(error)) {
    // Use fallback data
    return cachedData;
  }
  // Re-throw other errors
  throw error;
}
```

### 3. Register for Monitoring

```typescript
import { registerCircuitBreaker } from "@repo/circuit-breaker";

registerCircuitBreaker("my-service", myBreaker);
```

### 4. Don't Wrap Everything

Only wrap:
- External API calls
- Network requests
- Services with known reliability issues

Don't wrap:
- Local computations
- Database queries (use connection pooling instead)
- Critical synchronous operations

### 5. Test Failure Scenarios

```typescript
// Force circuit open for testing
clerkCircuitBreaker.forceState("OPEN");

// Test your fallback
// ... run your code ...

// Reset
clerkCircuitBreaker.forceState("CLOSED");
```

## Troubleshooting

### Circuit Keeps Opening

**Problem:** Service is genuinely struggling

**Solutions:**
1. Increase failure threshold
2. Check service health
3. Implement caching
4. Add load balancing

### Circuit Not Opening

**Problem:** Threshold too high or error detection wrong

**Solutions:**
1. Lower failure threshold
2. Fix `isFailure` function
3. Check error messages

### Stuck in HALF_OPEN

**Problem:** Service is flaky

**Solutions:**
1. Increase reset timeout
2. Increase success threshold
3. Add backoff strategy
