# Flaky Test Detection

This document describes the flaky test detection and quarantine system for the next-ship repository.

## Overview

Flaky tests are tests that pass or fail inconsistently without any code changes. They undermine confidence in the test suite and can block development. This system detects, tracks, and quarantines flaky tests to maintain CI reliability.

## Detection Methods

### 1. Automated Detection (CI)

The `flaky-test-detection.yml` workflow runs tests 3 consecutive times **daily at 2:00 AM AWST** (Western Australia Standard Time):

**Timezone Reference:**
- **2:00 AM AWST** (Perth, Western Australia) - Primary target time
- 6:00 PM UTC (GitHub Actions runs in UTC)
- 7:00 PM AWDT (Australian Western Daylight Time, when in daylight saving)

**What it does:**
- Runs all tests 3 times consecutively
- Compares results between runs
- If results vary (some pass, some fail), flags as potentially flaky
- Creates GitHub issues for tracking
- Uploads detailed test results as artifacts

**Note:** This workflow does **not** block PRs. It runs asynchronously to avoid slowing down development while still catching flaky tests.

### 2. Historical Analysis

Test results are stored as artifacts for 30 days, enabling:
- Pattern analysis over time
- Identification of intermittently failing tests
- Statistical detection of flakiness

### 3. Manual Identification

Developers can manually quarantine known flaky tests using the CLI.

## Quarantine System

### Configuration File

Known flaky tests are tracked in `.flaky-tests.json`:

```json
{
  "quarantinePolicy": {
    "maxRetries": 3,
    "autoQuarantineThreshold": 5,
    "maxQuarantineDuration": "30d"
  },
  "tests": [
    {
      "name": "test description",
      "file": "path/to/test.ts",
      "reason": "Why it's quarantined",
      "quarantinedAt": "2026-04-03",
      "issueUrl": "https://github.com/..."
    }
  ]
}
```

### Quarantine Policy

- **maxRetries**: Number of retry attempts before marking as failed
- **autoQuarantineThreshold**: Consecutive failures before auto-quarantine
- **maxQuarantineDuration**: Maximum time a test can stay quarantined (30 days)

## Commands

### Check for Flaky Tests

```bash
# Check if any tests are currently quarantined
pnpm test:flaky
```

### List Quarantined Tests

```bash
# View all quarantined tests with details
pnpm test:quarantined
```

### Quarantine a Test

```bash
# Add a test to quarantine
pnpm quarantine:add "test name" "path/to/test.ts" "Reason for quarantine"

# Example:
pnpm quarantine:add "database connection" "packages/db/__tests__/connection.test.ts" "Intermittent timeout in CI"
```

### Unquarantine a Test

```bash
# Remove a test from quarantine (when fixed)
pnpm quarantine:remove "test name"
```

## Handling Flaky Tests

### When a Flaky Test is Detected

1. **Immediate**: Investigate the test
2. **Short-term**: Quarantine if blocking CI
3. **Long-term**: Fix the root cause

### Common Causes of Flakiness

- **Race conditions**: Async operations without proper awaits
- **Timing issues**: Hardcoded timeouts, slow operations
- **Shared state**: Tests depending on each other
- **External dependencies**: Network, database, third-party services
- **Random data**: Tests using `Math.random()` or timestamps

### Best Practices to Avoid Flakiness

1. **Use proper async/await**: Always await promises
2. **Mock external services**: Don't depend on real APIs in tests
3. **Reset state**: Clean up before each test
4. **Avoid random data**: Use seeded random values
5. **Use explicit waits**: Don't assume timing

## GitHub Actions Workflows

### `flaky-test-detection.yml`

- **Trigger**: Daily at 2:00 AM AWST (scheduled)
- **Purpose**: Detect tests with inconsistent results
- **Action**: Creates GitHub issues when flaky tests found
- **Timezone**: 2:00 AM AWST = 6:00 PM UTC (runs at 18:00 UTC)

### `ci.yml` (Test job)

- **Trigger**: PR, push to main
- **Purpose**: Run tests with result tracking
- **Output**: JUnit XML reports uploaded as artifacts

## Troubleshooting

### Tests Pass Locally but Fail in CI

1. Check for timing differences
2. Verify all dependencies are installed
3. Check environment variable differences
4. Look for OS-specific behavior

### Intermittent Timeouts

1. Increase timeout thresholds
2. Check for race conditions
3. Verify resource cleanup
4. Consider mocking slow operations

### Database-Related Flakiness

1. Ensure test isolation
2. Reset database state between tests
3. Use transactions and rollbacks
4. Consider using test containers
