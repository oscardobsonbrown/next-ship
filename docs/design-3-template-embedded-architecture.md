# Design #3: Template-Embedded Documentation Architecture

## 1. Architecture Overview

### Core Philosophy
The documentation system treats docs as **first-class project assets** that ship with the template and evolve alongside the codebase. Unlike external docs or generated docs, this approach makes documentation part of the developer's mental model from day one.

### The Three Zones

The docs app is divided into three conceptual zones:

#### Zone 1: Template Reference (`/template/*`)
**Purpose**: Immutable documentation about the template itself  
**Access**: Read-only for template users  
**Update**: Managed by template maintainers via npm package updates

```typescript
// apps/docs/app/(content)/template/layout.tsx
// This zone is protected - users can read but not edit
export const templateZone = {
  source: 'node_modules/@next-ship/docs/template',
  editable: false,
  updateStrategy: 'npm-update',
  visualIndicator: 'template-badge'
};
```

#### Zone 2: Project Content (`/project/*`)
**Purpose**: User's custom documentation  
**Access**: Fully editable  
**Update**: Version controlled with user's project

```typescript
// apps/docs/app/(content)/project/layout.tsx
export const projectZone = {
  source: './content/project', // In user's repo
  editable: true,
  scaffolding: true, // Starts with template examples
  evolutionTracking: true
};
```

#### Zone 3: Auto-Generated (`/generated/*`)
**Purpose**: Documentation extracted from code  
**Access**: Read-only (regenerated on build)  
**Update**: Continuously synced with codebase

```typescript
// Generated from:
// - TypeScript types → API reference
// - Component props → Component docs
// - Database schema → Data model docs
// - Environment variables → Configuration docs
```

### The Scaffolding Process

When a user clones the template, the scaffolding system sets up their documentation:

```typescript
// apps/docs/lib/scaffolding.ts
interface ScaffoldConfig {
  projectName: string;
  description: string;
  features: ('auth' | 'billing' | 'api' | 'admin')[];
  teams: string[];
  existingDocs?: 'none' | 'markdown' | 'notion' | 'confluence';
}

class DocumentationScaffolder {
  async scaffold(config: ScaffoldConfig) {
    // Step 1: Copy starter content
    await this.copyStarterTemplates(config);
    
    // Step 2: Generate project-specific content
    await this.generateProjectOverview(config);
    await this.generateArchitectureDoc(config);
    await this.generateTeamDocs(config);
    
    // Step 3: Create customization hooks
    await this.createCustomComponents(config);
    await this.setupThemeOverrides(config);
    
    // Step 4: Initialize evolution tracking
    await this.initEvolutionTracker(config);
    
    // Step 5: Create migration guide
    await this.createMigrationGuide(config);
  }
}
```

### The Evolution System

The evolution system tracks the transformation from template docs to project docs:

```typescript
// apps/docs/lib/evolution.ts
interface EvolutionState {
  version: string;
  scaffoldDate: Date;
  milestones: Milestone[];
  customizations: Customization[];
  templateUpdates: TemplateUpdate[];
}

interface Milestone {
  type: 'first-edit' | 'first-page' | 'first-component' | 
        'team-docs' | 'api-docs' | 'custom-theme';
  date: Date;
  description: string;
  impact: 'low' | 'medium' | 'high';
}

class EvolutionTracker {
  // Track when user makes first customization
  async recordMilestone(type: Milestone['type']) {
    const milestone = {
      type,
      date: new Date(),
      description: this.getMilestoneDescription(type)
    };
    await this.saveMilestone(milestone);
    
    // Suggest next steps based on progress
    await this.suggestNextSteps();
  }
  
  // Track template updates that might affect user
  async checkTemplateUpdates() {
    const latestTemplate = await this.fetchLatestTemplate();
    const currentVersion = await this.getCurrentTemplateVersion();
    
    if (latestTemplate.version !== currentVersion) {
      const diff = await this.analyzeImpact(latestTemplate, currentVersion);
      
      return {
        available: true,
        version: latestTemplate.version,
        impact: diff.impact, // 'none', 'minor', 'major'
        changes: diff.changes,
        migrationRequired: diff.migrationRequired,
        autoMigratable: diff.autoMigratable
      };
    }
  }
  
  // Generate evolution report
  async generateReport(): Promise<EvolutionReport> {
    return {
      // How much has the user customized?
      customizationPercentage: await this.calculateCustomization(),
      // Which template features are still used?
      templateFeatureUsage: await this.analyzeTemplateUsage(),
      // Suggestions for next steps
      recommendations: await this.generateRecommendations(),
      // Evolution timeline
      timeline: await this.getTimeline()
    };
  }
}
```

## 3. What This Design Hides Internally

### Complexity Abstracted Away

| What Users See | What's Hidden |
|----------------|---------------|
| Simple MDX files | Content processing pipeline (remark, rehype, custom plugins) |
| `pnpm docs:dev` | Turborepo task orchestration, file watching, hot reload |
| Navigation config | Dynamic route generation, file-system routing abstraction |
| Auto-generated API docs | TypeScript parser, tRPC introspection, AST analysis |
| Template update notifications | Version diffing, impact analysis, merge conflict detection |
| Custom components | Component shadowing, build-time resolution, fallback handling |

### Internal Systems (Users Don't Touch)

### The Content Pipeline (Abstracted)

Users write simple MDX. The pipeline handles:

```typescript
// .internal/build-system/content-pipeline.ts
// ⭐ USERS NEVER SEE THIS

async function processContent(source: string): Promise<ProcessedContent> {
  return await unified()
    // 1. Parse MDX
    .use(remarkParse)
    .use(remarkMdx)
    
    // 2. Process template directives
    .use(remarkTemplateDirectives, {
      '<TemplateOnly>': (node) => {
        // Remove content that's template-specific
        if (isUserProject) return null;
        return node;
      },
      '<ProjectPlaceholder>': (node, config) => {
        // Inject project-specific content
        return fillPlaceholder(node, config);
      }
    })
    
    // 3. Enhance with smart features
    .use(remarkSmartLinks)        // Auto-link to known pages
    .use(remarkCodeTabs)          // Group related code blocks
    .use(remarkAutoImports)       // Import known components
    .use(remarkDiagramGeneration) // Generate diagrams from text
    
    // 4. Transform to React
    .use(rehypeReact, {
      components: resolveComponents()
    })
    
    .process(source);
}
```

### Component Shadowing System

Users can override template components without breaking updates:

```typescript
// .internal/build-system/component-resolver.ts
// ⭐ USERS NEVER SEE THIS

function resolveComponent(name: string): ComponentType {
  // 1. Check for user custom component
  const customPath = `./components/custom/${name}.tsx`;
  if (existsSync(customPath)) {
    return import(customPath);
  }
  
  // 2. Check for user config override
  const configOverride = docsConfig.components?.[name];
  if (configOverride) {
    return import(configOverride);
  }
  
  // 3. Fall back to template component
  return import(`./components/_template/${name}.tsx`);
}
```

## 4. Trade-offs of This Approach

### Advantages

| Advantage | Description |
|-----------|-------------|
| **Immediate Value** | Users have a working docs site from day 1, not an empty framework |
| **Contextual Learning** | Template docs are available while building, aiding understanding |
| **Progressive Disengagement** | Users can gradually replace template content as they grow |
| **Update Safety** | Template updates don't overwrite user content; clear separation |
| **Built-in Best Practices** | Template docs demonstrate good documentation patterns |
| **Evolution Tracking** | System helps users understand their documentation maturity |
| **Offline First** | Docs site works without external services (unlike Notion/Confluence) |

### Disadvantages

| Disadvantage | Mitigation |
|--------------|------------|
| **Initial Complexity** | Template appears larger with embedded docs | Clear documentation, optional "minimal" mode |
| **Monorepo Pollution** | Docs app takes space in user's repo | Docs are tree-shaken from production builds |
| **Template Update Noise** | Users must review template updates | Impact assessment shows relevance; can ignore minor updates |
| **Learning Curve** | Must learn docs system in addition to app | Progressive disclosure; can start with just MDX files |
| **Customization Overhead** | Rich customization system can feel heavy | Config is optional; works out of the box |
| **Build Time** | Generating docs adds to build | Parallel builds; static generation at build time |

### Comparison with Alternatives

| Approach | Template-Embedded | External Docs (Notion) | Generated Docs (Docusaurus) |
|----------|-------------------|------------------------|----------------------------|
| **Setup Time** | Minutes | Hours (create structure) | Hours (configure generator) |
| **Template Sync** | Built-in | Manual | N/A |
| **Customization** | High (code-level) | Medium (UI-limited) | Medium (config-limited) |
| **Team Collaboration** | Git workflow | Real-time editing | Git workflow |
| **Version Control** | Full history | Limited | Full history |
| **Offline Access** | Yes | No | Yes |
| **Maintenance Burden** | Low-Medium | Low | Medium |

### When This Approach Works Best

✅ **Ideal for:**
- Teams that value documentation as code
- Projects where docs need deep integration with codebase
- Organizations with multiple teams (engineering, product, design)
- Long-lived projects that will evolve significantly
- Teams comfortable with MDX and React

❌ **Not ideal for:**
- Simple projects with minimal documentation needs
- Teams that strongly prefer WYSIWYG editing
- Projects where documentation is primarily customer-facing help
- Organizations already invested in Confluence/Notion with no migration appetite

### The "Escape Hatch"

If users outgrow the embedded docs:

```bash
# Export to standalone documentation
pnpm docs:export --format=docusaurus --output=../my-docs-site

# Or migrate to Notion
pnpm docs:export --format=notion --notion-token=xxx

# Or keep the build, ditch the source
pnpm docs:archive
```

## Summary

The Template-Embedded Documentation architecture treats documentation as a living project asset that ships with the template and evolves alongside the codebase. By separating template reference content (read-only, updated via npm) from project content (fully editable, version controlled with the project), it provides:

1. **Immediate productivity** with pre-built structure and examples
2. **Safe customization** through a hook-based configuration system
3. **Clear evolution path** tracked by an intelligent evolution system
4. **Graceful template updates** with impact analysis and migration assistance
5. **Full ownership** - the documentation truly belongs to the user's project

This approach optimizes for the template user experience by making documentation part of what they clone and customize, not an afterthought or external dependency.
